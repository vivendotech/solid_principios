Se você é programador a mais de uns dois anos você já deve ter percebido que um software com um bom banco de códigos faze toda a diferença. Provavelmente porque você já foi atrasado por um código ruim.[2] Geralmente no inicio do projeto o código é desenvolvido super rápido porém conforme o tempo passa ele começa a ser desenvolvido a passos de tartaruga. Isso acontece porque nenhuma mais mudança é trivial. A cada nova mudança implementada se quebra duas ou três partes do código. Nesta série sobre refatoração e boas práticas de desenvolvimento começamos[2] discutindo sobre os valores de desenvolvimento Agile . Nesse post iremos nos aprofundar nos sintomas de um código que cheira mal que irá ser apresentado quando ele estiver apodrecendo.

O Sintoma da Rigidez

O sintoma de que um software está rígido é percebido quando o sistema fica difícil de ser mudado, mesmo de uma forma simples. Em um designe rígido uma simples mudança gera uma cascata subsequentes de mudanças em módulos dependentes. A métrica geralmente usada para detectar rigidez é a quantidade de módulo que precisam ser mudados dado uma mudança de um módulo. [1]

Sintoma da fragilidade

O sintoma da fragilidade é encontrado quando uma mudança feita no código faz com que vários lugares diferentes do código parem de funcionar. Um fator agravante é quando os novos problemas aparecem em áreas diferentes da mudança. [1]

Sintoma da Imobilidade

O software apresenta o sintoma da imobilidade quando uma parte do código que pode ser movidas para ser utilizada em outra parte do código não é porque não pode ser movida. Geralmente não pode-se separar o código porque é muito arriscado.[1]

Sintoma da Viscosidade

O sintoma da viscosidade se apreenta de duas formas : viscosidade no software e viscosidade no ambiente. O sintoma da viscosidade no software é encontrado quando existe mais de uma forma de criar uma mudança. Uma das formas de criar mudanças é respeitando o designe e as outras alternativas não respeitão. Quando é mais facil fazer a mudança desrespeitando o desgine mais viscoso o designe é. [1]

A viscosidade dos ambientes de desenvolvimento se devem quando os ambientes de trabalho são lentos e ineficientes. Por exemplo, se um sistema leva um grande tempo para compilar os designer vão fazer mudanças que não levam tanto tempo para recompilar mesmo se essas mudanças não preservem o designe.[1]

Complexidade desnecesária

O software apresenta o sintoma da complexidade desnecessária quando o programador implementa facilidades e expansões do código para mudanças que ainda não foram necessárias. O problema acontece porque muitos planos de contigente são criados mas nunca usados. Geralmente o designe carrega um peso desses elementos desnecessários. Essa facilidades e expansões tornam o software mais complexo e difícil de entender.[1]

Repetição desnecesária

O sintoma da repetição desnecessária está atribuída a códigos redundantes no software. Quando um software possui códigos redundantes, modificar o sistema se torna uma tarefa mais arduosa. Bugs podem ser encontrado em uma repetição do código redundante e então deve ser encotnrado a cada repetição do código que pode ser um pouco diferente da repetição anterior. Assim o concerto pode ser que não seja o mesmo em cada repetição. O outro problema é decorrente é porque de desenvolvedores estão perdendo de fazer uma abstração.[1]

Sintoma de opacidade

O sintoma da opacidade está atrelado a tendencia que os códigos tem de se tornarem mais difíceis de entender. Os códigos podem ser escritos em uma forma clara e expressiva ou em uma forma opaca e enrolada. Codigos com o tempo tem a tendencia de se tornar mais opaco. Isso acontece porque os códigos recém desenvolvidos estão fresco na memória do programar. Com o passar do tempo ele vai esquecendo e o código vai se tornando opaco. Esse problema seria facilmente evitados se o desenvolvedor se coloca-se no lugar das pessoas que vão ler o seu código. Isso é uma boa prática porque em algum futuro essa pessoa pode ser você mesmo.
